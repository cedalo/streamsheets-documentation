Version: Mosquitto 3.1

Migration Information

Adding new features can require to adapt the existing configuration of the Pro Mosquitto and MMC setups. This page documents possible configuration and setup changes. While updating your on-premise setups please refer to this document. Cloud customers are updated automatically.

  • Documentation: Visit the respective docs to learn how to configure and utilize the new features available in version 2.9. The documentation provides detailed instructions and insights to help you seamlessly integrate these features into your existing setup.
  • License Update: If your current license does not cover the version features you are upgrading to, it is essential to update your license to access the full capabilities of the new release. The old license will still work for all existing features.
  • Cluster updates: If you are updating a cluster, make sure to update each Mosquitto node one by one, with the leader being the last node to be updated.

Should you encounter any issues during the migration process or if you have any questions regarding the new features or license updates, please do not hesitate to reach out to us. Our support team is ready to assist you at support@cedalo.com.

3.1 Update

Release 3.1 introduces major improvements to the bridge functionality. This includes the addition of new bridge types, the introduction of a dedicated bridge user interface, and full support for bridges in clustered environments. These enhancements required internal changes to the bridge handling architecture to ensure robust performance and scalability.

Breaking Changes

Bridges

  • JSON config files of bridges that are configurable via the Cedalo MQTT Platform should not be edited manually anymore (see for further information). For Single Node setups they are solely used as backup files and managed by the broker executable. In Cluster Setups the JSON file is not needed anymore as all changes made in the UI are part of the cluster synchronized data, which already gets written to file. See Migration section for required changes about the configuration of the respective plugins.

MongoDB

  • The config file schema has changed in a way that further aligns it with other plugins to simplify the process for our users as they encounter similar patterns when configuring different bridges.

Migration

Update current installation

  • Docker environments: Please update your docker-compose.yml file(s) to use the new images:
    • For Pro Mosquitto: registry.cedalo.com/mosquitto/mosquitto:3.1
    • For Cedalo MQTT Platform (depending on your license):
      • registry.cedalo.com/platform/business:3.1
      • registry.cedalo.com/platform/enterprise:3.1
    • Make sure to restart all instances after modifying the docker-compose.yml file(s):
      • docker compose up -d (Single Node setups)
      • ./start.sh up -d (Cluster Setups)
  • Alternative Download Option: If you prefer, you can also download the new .rpm package from the Cedalo MQTT Platform for a manual update.

MongoDB (Single Node Migration - on-prem)

  1. Proceed with migrating your configuration to the new version (see Breaking Changes for MongoDB) section.
  2. Rename the JSON config file to be named: MongoDB-Bridge.json.
  3. Done.

MongoDB (Cluster Migration - on-prem)

  1. Proceed with migrating your configuration to the new version (see Breaking Changes for MongoDB) section.
  2. Copy all contents of the mongodb-bridge.json file.
  3. In the Cedalo MQTT Platform direct to the Bridges menu.
  4. Find the MongoDB-Bridge entry and click on the edit icon.
  5. Click on the Edit JSON tab.
  6. Paste the content into the built-in JSON editor and click on Save Configuration.
  7. The platform should now display a success message. The config is automatically under the hood sychronized accross the cluster.
  8. You can keep the mongodb-bridge.json file as a personal backup or delete it, as it is not needed by the cluster anymore (see Breaking Changes).
  9. Done.

3.0 Update

The 3.0 release introduces a completely new UI for the Pro Mosquitto broker and includes some breaking changes. To ensure a smooth transition, please follow the steps outlined below.

Breaking Changes

  • Management Center users and user groups must be set up again.
  • REST API of the Platform has breaking changes to the existing MMC REST API.
  • CA certificates & API tokens are not transferred from the old UI MMC to the new Platform UI.

These breaking changes only affect the UI. A full migration of the existing Mosquitto broker is supported.

Single Node Migration

Mosquitto & Platform

  1. Note admin credentials used by the MMC connections to the broker. These are required to connect the broker to the platform.
  2. Download the new 3.0 setup and stop the existing installation with docker compose down.
  3. Backup your current setup folder.
  4. In the new setup directory, copy the platform folder, license folder, and the new docker-compose.yml file into your existing setup folder.
    • If you have adopted further port bindings, please add them in just copied docker-compose.yml (e.g. below the mosquitto service).
    • Optional: If you want to keep the old MMC, add the MMC service section to the new docker-compose.yml file to enable parallel usage of the old and new UIs.
  5. Add the license file to the license folder.
  6. Update the mosquitto.conf file by adding the new topic tree plugin:
    # Topic Tree plugin
    plugin /usr/lib/cedalo_topic_tree.so
  7. Start the updated setup:
    docker compose up -d
    Run this command at the same level as the docker-compose.yml file.

Re-establish the connection to the Mosquitto broker

  1. Open the new Cedalo MQTT Platform UI via localhost:3000.
  2. Create a new user.
  3. In the connection edit enter the connection details as configured in the existing MMC connection
    • Before saving the changes you can test the connection to the Mosquitto broker by clicking on Test Connection

Only Mosquitto

To update from 2.9 to 3.0 you can simply update the existing image in docker or install the new rpm package for RHEL. The update is less complicated, because no new UI is introduced, if only Mosquitto is updated to 3.0. This could be the case if you decide to update all your brokers and connect multiple instances of the brokers to the same Cedalo Platform. In this case only one Platform installation is needed.

Before starting it makes always sense to make some backups of your current setup folder.

Docker Update

Find your docker compose file and update the existing image. The new image path is:

image: registry.cedalo.com/mosquitto/mosquitto:3.0

Now execute:

docker compose down
docker compose up -d

or use the existing 'stop', 'update' and 'start' scripts as described in your readme file.

RPM Update

First stop mosquitto:

systemctl stop mosquitto-premium

Then delete the old Mosquitto:

dnf remove mosquitto-premium

Install new version:

cd <directory with 3.0 rpm package>
dnf install <3.0 rpm package>

Make mosquitto execute on startup and start the service:

systemctl enable mosquitto-premium
systemctl start mosquitto-premium

Post-Update Steps

Once everything is working correctly, you can remove your backed-up files (though it's recommended to wait a few days before doing so).

The migration is now complete.

Cluster Migration

Mosquitto & Platform

To migrate a cluster, update one server at a time, starting with the two follower nodes. Only copy the platform folder if updating the node where the platform is installed.

  1. Download the new 3.0 setup.
  2. Note down admin credentials for MMC connections to the broker and your cluster connection details.
  3. Select a follower node and:
    • Stop the existing installation with docker compose down.
    • Backup the current setup.
  4. Update the docker-compose file:
    • Locate the mosquitto service and update the image to the 3.0 version.
    • If this server also runs the Management Center:
      • Remove the existing management-center service and all related settings from the docker-compose.yml.
        • Optional: To keep it temporarily, change its IP address in docker-compose.yml to ipv4_address: 172.20.4.1.
      • Copy the new platform service from the 3.0 setup into the existing docker-compose.yml file, ensuring the hierarchy is correct. Also copy the platform folder from the new setup into the current server folder.
      • Delete the current platform database file located at server1/platform/data/mosquitto-platform.sqlite3.
  5. Update the mosquitto.conf file by adding the new topic tree plugin:
    # Topic Tree plugin
    plugin /usr/lib/cedalo_topic_tree.so
  6. Start the updated setup:
    docker compose up -d
    After updating the first follower, proceed with the second follower, and finally update the leader node.

Cluster Recreation

  1. Open the new Cedalo MQTT Platform UI via localhost:3000.
  2. Create a new user.
  3. Enter the connection details of one node as configured in the existing MMC connection.
    • Upon successful connection, the existing cluster will be detected, and you can automatically add the other nodes.
  4. Set the public connection details to reach the cluster, then click Save.

Only Mosquitto

For updating Mosquitto in a cluster environment, follow the single node update process described above for each node in your cluster. The process is as follows:

  • Create backups of your setup folders on all nodes.
  • Identify your current leader node.
  • Update follower nodes first, one at a time.
  • Update the leader node last.

Only proceed to the next node when you've confirmed the current node is working correctly within the cluster. The migration is now complete.

Single Node

This is the Docker Migration guide to upgrade from version 2.6 to 2.7.

Docker Compose

  1. Add the following environment variable to the management center:
CEDALO_MC_PROXY_HOST: 0.0.0.0
  1. You may remove the following variable from the docker-compose as it's no longer being used:
CEDALO_MC_PLUGIN_DIR: /management-center/plugins
  1. Optionally: A new way of loading MMC plugins was added. The file "plugins.json" can now be used to manually configure the plugins. The config.json loads the content of the file on start if the environment variable "CEDALO_MC_PLUGIN_LIST_PATH: management-center/config/plugins.json" is set. If no file is found an error message is shown and the config.json file will be read. If the file is found, the config.json file plugin list will be overwritten on start.

Specify some specific set of plugins to load via the CEDALO_MC_PLUGIN_LIST_PATH environment variable:

CEDALO_MC_PLUGIN_LIST_PATH: management-center/config/plugins.json"

Leaving everything in a single docker-compose file

If you want to leave everything similar to the old setups and use a single docker-compose.yml file than make sure to make the following changes to this file to upgrade to 2.7.

Note that in the future we will not support this setup and will not provide migration guides for a single `docker-compose.yml`` file.

  1. Change mosquitto images to registry.cedalo.com/mosquitto/mosquitto:2.7. Change MMC image to registry.cedalo.com/mosquitto/management-center:2.7

  2. Update your licenses in management-center/config, mosquitto1/license, mosquitto2/license, and mosquitto3/license.

  3. Add the following environment variable:

CEDALO_MC_PROXY_HOST: 0.0.0.0
  1. Make sure you have CEDALO_MC_PROXY_CONFIG_CLUSTERS from your old setup.
CEDALO_MC_PROXY_CONFIG_CLUSTERS: /management-center/config/clusters.json

Technically, you don't need to explicitly define it as long as you set CEDALO_MC_DATA_DIRECTORY_PATH (because in this case clusters.json will be looked up in $CEDALO_MC_PROXY_CONFIG_CLUSTERS/clusters.json), but it's still a good idea to mention it explicitly, especially if you had clusters.json stored in a non-default location.

  1. If you are going to use Audit Trail, add the following variables:
CEDALO_HOST_NAME: ${CEDALO_HOST_NAME}
CEDALO_IP_ADDRESS: ${CEDALO_IP_ADDRESS}

CEDALO_MC_PROXY_CONFIG_AUDIT_TRAIL: /management-center/config/audit.json

Note that if you have CEDALO_MC_DATA_DIRECTORY_PATH specified, you don't have to set CEDALO_MC_PROXY_CONFIG_AUDIT_TRAIL as well, but specifying it explicitely is a good idea.

CEDALO_HOST_NAME and CEDALO_IP_ADDRESS should be provided before running docker-compose (they will be automatically set when running with start.sh or start.bat). These variables will display hostname and ip address of the mmc application in the syslog entries.

  1. If you want to use audit-trail you should additionally specify the following mount if you want your container to direct the syslogs into the hostmachine's /dev/log unix socket:
volumes:
    # other volumes here...
    - /dev/log:/dev/log
  1. You may remove the following variable from the docker-compose as it's no longer being used:
CEDALO_MC_PLUGIN_DIR: /management-center/plugins
  1. Optionally: A new way of loading MMC plugins was added. The file "plugins.json" can now be used to manually configure the plugins. The config.json loads the content of the file on start if the environment variable "CEDALO_MC_PLUGIN_LIST_PATH: management-center/config/plugins.json" is set. If no file is found an error message is shown and the config.json file will be read. If the file is found, the config.json file plugin list will be overwritten on start.
CEDALO_MC_PLUGIN_LIST_PATH: management-center/config/plugins.json"

Features

The new version added new features, which are not active by default. As usual, features are implemented via plugins, which can be turned off and on in the configuration files of the Management Center and Mosquitto. Loading plugins, which are not used can increase resources needed. This is why certain plugins need activation. To active new 2.7 features please see their documentation.
